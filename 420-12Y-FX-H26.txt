1 - MISE EN CONTEXTE
1.1 - Objectif
Cette activité de synthèse est l’occasion pour vous de démontrer, en l’espace de trois semaines, les compétences acquises durant vos trois années d’études. Le travail demandé consiste à développer une application Web, appuyée par une base de données, de type Help Desk permettant la gestion complète des incidents (tickets) dans une organisation. L’application, nommée GarneauSupport?? doit permettre ; 
•	La gestion des comptes client et techniciens
•	La gestion des catégories d’incidents
•	L’assignation à des techniciens
•	La gestion d’un cycle de vie structuré
•	La communication entre utilisateurs et techniciens
•	La résolution et la fermeture des incidents

2 – CONTRAINTES DE RÉALISATION
2.1 – Technologies à utiliser
Liste des technologies à utiliser pour réaliser l’application
•	Votre application doit utiliser un Backend Node.js avec Express.
•	Votre Frontend doit utiliser Vue.js (version 3) avec l’API Composition.
•	Votre Frontend et votre Backend doivent être dans deux projets différents (deux dossiers différents) et communiquer via des requêtes HTTP (utilisez le modèle REST).
•	Vous devez obligatoirement utiliser Bootstrap 5 (version Sass) pour le style de votre application. Vos propres styles devront aussi utiliser Sass.
•	Votre application doit utiliser une base de données SQL avec le moteur MySQL ou 
•	Votre Backend doit utiliser l’ORM Sequelize pour communiquer avec votre base de données.
Note importante concernant l'utilisation de l'intelligence artificielle (IA)
Conformément à la Politique institutionnelle d’évaluation des apprentissages (PIEA) du Cégep Garneau, l’utilisation de l’intelligence artificielle (ex. : ChatGPT, Copilot, etc.) dans la réalisation de ce travail est considérée comme une forme de plagiat ou de tricherie. Tout recours à ces outils sans autorisation constitue une infraction à la PIEA et peut mener à une déclaration de fraude.[MV1.1]
2.2 – Utilisation de Git
Le développement de votre application doit être fait dans un dépôt sur gitlab.com.
•	Le dépôt doit être privé ;
•	Vous devez ajouter l’utilisateur @dept-info-garneau en tant que « Developer »;
•	Il n’y a pas de stratégie de versionnage imposée. La seule contrainte est d’avoir un code à jour sur la branche main avant la date de remise pour chaque semaine de travail. 
2.3 – Contraintes techniques
•	Le système de navigation doit être adapté aux différents rôles d’utilisateur.
•	Toutes les pages HTML générées doivent être valides d’après le W3C.
•	Le CSS doit être généré par Sass.
•	La mise en page doit bien s’adapter à divers types d’appareils (responsive).
•	Tous les documents produits dans le cadre de ce travail doivent s'afficher correctement avec les navigateurs Web Firefox et Chrome.
Évolution des exigences[MV2.1][GD2.2][MV2.3]
Dans un contexte de développement agile, il est important de comprendre que les besoins du client peuvent évoluer au cours du projet.
Ainsi, durant les sprints de développement, le client (enseignant) pourrait :
•	Demander des ajustements mineurs 
•	Proposer des améliorations fonctionnelles 
•	Clarifier ou modifier certaines règles métier 

Ces changements peuvent survenir :
•	Pendant un sprint (de façon encadrée) 
•	Entre deux sprints (lors des suivis) 

Note générale
Toute action d’ajout, de modification ou de suppression doit générer une rétroaction visuelle claire pour l’utilisateur. Les suppressions doivent être confirmées avant d’être exécutées. Par ailleurs, les éléments supprimés ne sont pas effacés physiquement de la base de données, mais marqués comme archivés et deviennent ainsi inaccessibles dans l’application.

3.5 – Gestion des délais de traitement[MV17.1]
Afin d’assurer un suivi efficace des incidents, chaque ticket doit respecter un SLA (Service Level Agreement) basé sur sa priorité.
Le SLA définit :
•	le délai maximal de prise en charge (assignation) 
•	le délai maximal de résolution 

Ces délais sont calculés à partir de la date de création du ticket.
Délais selon la priorité
Priorité	Délai de prise en charge	Délai de résolution
Basse	24 heures	72 heures
Moyenne	8 heures	24 heures
Haute	2 heures	8 heures
Critique	30 minutes	4 heures

Règles à respecter
•	Un ticket doit être assigné à un technicien dans le délai de prise en charge.
•	Un ticket doit être résolu dans le délai de résolution.
•	Si un délai est dépassé :
o	le ticket est considéré en retard
o	un indicateur visuel doit être affiché dans l’interface (ex : couleur, badge)

3.5 – Sécurité de l’application
Votre application doit respecter les contraintes de sécurité suivantes :
•	Vos mots de passe doivent être hachés en utilisant la librairie bcrypt.
•	Vous devez valider les données du côté serveur et du côté client.
•	Vous devez retourner les codes d'erreurs appropriés dans les cas suivants :
•	Accès à une ressource inexistante : code 404. 
•	Accéder à une ressource protégée sans être authentifié : code 401.
•	Accéder à une ressource protégée sans les permissions nécessaires : code 403.
•	Erreurs de validation : code 422.
•	Erreur de conflit : code 409.
•	Erreur serveur : code 500.
